# Schema Linking: How We Augment BIRD Data for Text-to-SQL

## The Problem

Raw database schemas are overwhelming for LLMs. A database like `financial` has 8 tables, 55 columns, and dozens of foreign key relationships. Dumping the full DDL into a prompt wastes context and confuses the model with irrelevant tables and columns.

BIRD benchmark provides gold SQL queries and evidence hints, but does **not** provide per-question schema context. The agent must figure out which tables and columns are relevant on its own.

## Our Solution: Perfect Schema Linking with Evidence Enrichment

We extract from [InsightXpert](https://github.com/NachiketKandari/InsightXpert) a two-stage pipeline that produces **pruned, semantically-enriched schema context** for each task.

### Stage 1: Evidence-Backed Database Profiling

For each database, InsightXpert's profiler generates rich column metadata:

1. **Statistical profiling** — For every column: data type, cardinality, null count, min/max values, and 20 sample distinct values via SQL queries.

2. **Evidence unification** — All per-question evidence hints from the BIRD benchmark are consolidated into a single database-level reference document using an LLM. This captures domain knowledge like "CDSCode is a 14-digit identifier composed of County, District, and School codes."

3. **LLM-generated column descriptions** — Each column gets a 1-2 sentence business-semantic description generated by an LLM, informed by both the column statistics and the unified evidence. This produces descriptions like:
   > The `CustomerID` column contains unique integer identifiers for each customer, serving as the primary key for the dataset.

   Instead of the raw stat-only version:
   > The "CustomerID" column contains integers ranging from 3 to 53,314 with no null values.

These profiles are pre-computed and stored as `profile.json` / `schema.json` per database.

### Stage 2: Gold SQL Parsing + Pruned Schema Rendering

For each task, given the gold SQL query:

1. **SQL parsing** (via [sqlglot](https://github.com/tobymao/sqlglot)) — Extract the exact tables, columns, aliases, and string literals referenced in the gold SQL.

2. **Column resolution** — Unqualified columns (no table prefix) are resolved against the database schema by matching column names across all tables.

3. **Join path completion** — For every table referenced in the gold SQL:
   - All primary key columns are included (even if not explicitly in the SELECT)
   - Foreign key columns pointing to other referenced tables are included
   - This ensures the agent has the information needed to write correct JOINs

4. **Pruned schema rendering** — Only the relevant tables and columns are rendered, annotated with:
   - Data type and PK/FK markers
   - The evidence-enriched column description from Stage 1
   - Sample values for low-cardinality columns
   - A "Joinable columns" section listing FK relationships between included tables

### Example Output

For a question about California school enrollment:

```
Table: "frpm"
  Columns:
    - "CDSCode" (TEXT, PK, FK -> schools.CDSCode): The "CDSCode" column contains
      the 14-digit unique identifier for California schools, composed of
      concatenated County, District, and School codes.
    - "Enrollment (K-12)" (REAL): Total K-12 enrollment count for the school.
    - "FRPM Count (K-12)" (REAL): Number of students eligible for Free or
      Reduced Price Meals in K-12.

Table: "schools"
  Columns:
    - "CDSCode" (TEXT, PK): Unique school identifier.
    - "School" (TEXT): Name of the school.
    - "County" (TEXT): County where the school is located.

Joinable columns:
  frpm.CDSCode -> schools.CDSCode
```

Compare this to the raw DDL which would include 15+ irrelevant columns from `frpm` and 20+ from `schools`.

## Why This Matters for RL Training

- **Without schema linking, RL is infeasible.** The probability of generating correct SQL from a 200-column schema is near zero, making reward signals too sparse.
- **With schema linking, the hard part remains.** The agent still must solve logical reasoning: correct JOINs, WHERE vs HAVING, subqueries, aggregation, NULL handling, CASE WHEN, and ORDER BY.
- **State-of-the-art models achieve 60-75% on BIRD even with perfect schema linking.** There is substantial room for RL-driven improvement.

## Files in This Directory

| File | Purpose |
|------|---------|
| `__init__.py` | Package docstring (offline-only declaration) |
| `models.py` | Pydantic models for schema and profile data structures |
| `db.py` | Minimal read-only SQLite wrapper |
| `schema_extractor.py` | Extract table/column metadata via PRAGMA queries |
| `stats_collector.py` | Collect per-column statistics (cardinality, samples, etc.) |
| `perfect_linker.py` | Core: parse gold SQL, resolve columns, add join paths, render pruned schema |
| `build_linking.py` | CLI orchestrator: load profiles, run perfect linking for all tasks |

## Usage

```bash
# Regenerate schema_linking.json using evidence-enriched profiles from InsightXpert:
python3 -m schema_linking.build_linking \
  --profiles-dir /path/to/insightxpert/profiles/mini_dev \
  --evidence

# Without evidence (stats-only, no LLM descriptions):
python3 -m schema_linking.build_linking \
  --profiles-dir /path/to/insightxpert/profiles/mini_dev

# From scratch (no pre-computed profiles, profiles databases on the fly):
python3 -m schema_linking.build_linking
```

## Data Flow

```
InsightXpert profiler (offline, LLM-powered)
  |
  v
profiles/mini_dev/{db_id}/evidence/profile.json  (enriched column descriptions)
profiles/mini_dev/{db_id}/schema.json             (database structure)
  |
  v
schema_linking/build_linking.py --evidence
  |  Loads profiles, iterates all 200 tasks
  |  For each task: parse gold SQL -> extract tables/columns -> add FK/PK bridges -> render
  v
data/schema_linking.json  (200 entries, each with pruned + enriched schema_text)
  |
  v
server/bird_environment.py  (loads at startup, serves to agents via observations)
```
